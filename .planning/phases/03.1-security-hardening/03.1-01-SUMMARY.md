# Phase 03.1 Plan 01: Security Headers and CSP Summary

**Static CSP + five HTTP security headers in next.config.ts, plus server-only guard on dal.ts — closing the highest-priority security gaps before deployment**

## Accomplishments
- Added `import 'server-only'` as first line in `src/lib/dal.ts` — prevents accidental import in client components, session verification logic stays server-only
- Rewrote `next.config.ts` with `async headers()` delivering X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, and Content-Security-Policy on all routes
- CSP configured correctly: allows Supabase API and Storage (connect-src + img-src), Tailwind v4 unsafe-inline styles, local Heebo font (no CDN), unsafe-eval in dev only for Turbopack HMR
- HSTS omitted from dev (prevents localhost lockout), included in production with 2-year max-age + preload
- Both X-Frame-Options and CSP frame-ancestors set for full clickjacking protection across all browsers
- Build passes cleanly; headers verified with `curl -I http://localhost:3000/login`

## Files Created/Modified

## Decisions Made
- **next@16.1.6 is already the latest stable version** — `npm install next@latest` confirmed no newer version exists; 16.0.10 patch was for the 16.0.x line and carried forward into 16.1.x
- **Static CSP** over nonce-based: nonces would force every admin page to dynamic rendering (losing static optimization); admin panel has no third-party scripts so static CSP + unsafe-inline is the correct tradeoff
- **No Google Fonts CDN in CSP**: Heebo is loaded via `next/font/google` which downloads and serves fonts locally — `fonts.googleapis.com` and `fonts.gstatic.com` are never hit at runtime
- **unsafe-eval in dev only**: Turbopack HMR requires eval; removed in production by `isDev` guard
