## Summary

This is an admin-only, single-user panel (Sharon only) managing employee PII (IDs, addresses, passport numbers). The system is built on Next.js 16.1.6 with Supabase and deployed to Vercel. A full codebase audit was performed to identify the current security posture and gaps.

**Current state is solid but incomplete.** The foundation is correct: all Server Actions call `verifySession()`, Zod validates all input, Supabase RLS is enabled on all tables, and `createAdminClient()` is server-only guarded. The gaps are: no security headers in `next.config.ts`, no rate limiting on the login route, no CSP, and a critical unpatched Next.js vulnerability (December 2025 CVEs).

**Critical alert:** Next.js 16.1.6 is affected by CVE-2025-55183 (Source Code Exposure) and CVE-2025-55184/CVE-2025-67779 (DoS). Fixed in Next.js 16.0.10 — but 16.1.6 requires upgrade to the patched 16.x version (16.1.x line fix is `16.1.0-canary.19` or stable 16.x equivalent). The version table shows 16.0.x is fixed in 16.0.10, so 16.1.6 likely has a patch too — this must be verified and the package upgraded.

**Primary recommendation:** Three tasks in priority order: (1) Upgrade Next.js to latest patched version, (2) Add security headers + CSP in `next.config.ts`, (3) Add in-memory rate limiting for the login Server Action.

### RLS Policy Assessment

**Current policy model:** Phase 1 "permissive authenticated" — any authenticated user can SELECT/INSERT/UPDATE any row.

**For the admin-only context this is ACCEPTABLE** because:
- The only way to authenticate is via Supabase Auth + `verifySession()` in middleware
- Sharon is the only user
- No public-facing routes exist
- All mutations go through Server Actions that call `verifySession()` first

**The one genuine RLS gap:** `user_permissions` table currently has policies allowing any authenticated user to INSERT/UPDATE permissions for ANY user. This is a Broken Access Control risk if any second user ever gets added. Tightening this policy to check caller's `is_admin` status would be the correct hardening.

**Proposed tighter policy for `user_permissions`:**
```sql
-- Instead of: WITH CHECK (true)
-- Use: WITH CHECK (
--   EXISTS (SELECT 1 FROM users WHERE auth_user_id = auth.uid() AND is_admin = true AND deleted_at IS NULL)
-- )
```

**What:** Add `headers()` async function to `next.config.ts` returning all security headers for all routes.

**Source:** [Next.js official docs — headers()](https://nextjs.org/docs/app/api-reference/config/next-config-js/headers) — HIGH confidence


### Pattern 2: Static CSP for Next.js 16 + Supabase + Tailwind

**What:** CSP policy that allows Supabase API calls, Heebo font from Google Fonts (if used via CDN), Tailwind inline styles.

**Source:** [Next.js CSP Guide](https://nextjs.org/docs/app/guides/content-security-policy) — HIGH confidence (fetched Feb 2026)

**Key insight for this project:** Use static CSP (not nonce-based) to avoid forcing all admin pages to dynamic rendering. Since the admin panel has no third-party scripts, `'unsafe-inline'` for styles is needed for Tailwind v4 + shadcn/ui.

```typescript
// CSP for ChemoSystem — no third-party scripts, Supabase connect, Google Fonts
const isDev = process.env.NODE_ENV === 'development'

const cspHeader = [
  "default-src 'self'",
  // Next.js scripts need 'unsafe-inline' because scripts are inline-injected
  // In production without nonces, 'unsafe-inline' is the pragmatic choice
  `script-src 'self' 'unsafe-inline'${isDev ? " 'unsafe-eval'" : ''}`,
  // Tailwind v4 + shadcn/ui inject inline styles
  "style-src 'self' 'unsafe-inline' https://fonts.googleapis.com",
  // Google Fonts CDN (if using Heebo via CSS @import)
  "font-src 'self' https://fonts.gstatic.com data:",
  // Supabase API calls
  `connect-src 'self' ${process.env.NEXT_PUBLIC_SUPABASE_URL ?? ''}`,
  // Employee photos from Supabase Storage
  `img-src 'self' blob: data: ${process.env.NEXT_PUBLIC_SUPABASE_URL ?? ''}`,
  "object-src 'none'",
  "base-uri 'self'",
  "form-action 'self'",
  "frame-ancestors 'none'",
  "upgrade-insecure-requests",
].join('; ')
```

**Important note:** If Heebo font is loaded via `next/font/google` (local serving), the `fonts.googleapis.com` and `fonts.gstatic.com` sources are NOT needed. Verify in `layout.tsx`.

### Pattern 3: In-Memory Rate Limiting for Login Server Action

**What:** Simple Map-based counter in the login Server Action, using client IP from request headers.

**Source:** Multiple verified patterns — MEDIUM confidence (pattern is well-established, but in-memory approach has limitations)

**Key architectural constraint:**
- Next.js middleware (`src/proxy.ts`) runs on Edge Runtime — Edge Runtime does NOT persist in-memory state across requests. A `Map` defined in a middleware module file is reset on every request invocation at the edge.
- Server Actions run in the Node.js runtime — module-level state (a `Map` singleton) IS persisted across requests within the same Vercel serverless function instance.
- **Therefore: Rate limiting MUST be implemented inside the login Server Action (Node.js runtime), NOT in the proxy middleware (Edge Runtime).**

```typescript
// src/actions/auth.ts — rate limiting module-level singleton
// Works in Node.js runtime (Server Actions are always Node.js)
// WARNING: resets on cold start, cannot share across multiple instances
// Acceptable for: single-user admin panel, single Vercel instance

type RateLimitEntry = { count: number; resetAt: number }
const loginAttempts = new Map<string, RateLimitEntry>()

const RATE_LIMIT_MAX = 5       // max attempts per window
const RATE_LIMIT_WINDOW = 60_000  // 1 minute in ms

function checkRateLimit(ip: string): boolean {
  const now = Date.now()
  const entry = loginAttempts.get(ip)

  if (!entry || now > entry.resetAt) {
    loginAttempts.set(ip, { count: 1, resetAt: now + RATE_LIMIT_WINDOW })
    return true // allowed
  }

  if (entry.count >= RATE_LIMIT_MAX) {
    return false // blocked
  }

  entry.count++
  return true // allowed
}
```

**Getting client IP in Server Action:**
```typescript
import { headers } from 'next/headers'

export async function login(prevState, formData) {
  const headersList = await headers()
  // On Vercel, client IP is in x-forwarded-for
  const ip = headersList.get('x-forwarded-for')?.split(',')[0]?.trim()
    ?? headersList.get('x-real-ip')
    ?? 'unknown'

  if (!checkRateLimit(ip)) {
    return { error: 'יותר מדי ניסיונות התחברות. נסה שוב בעוד דקה.' }
  }
  // ... rest of login logic
}
```

### Pattern 4: CSRF Protection — Already Built-In

**Source:** [Next.js Data Security Guide](https://nextjs.org/docs/app/guides/data-security) — HIGH confidence (fetched Feb 2026 official docs)

Server Actions in Next.js have **two layers of built-in CSRF protection**:
1. **POST-only:** Server Actions only accept `POST` method — prevents CSRF via GET
2. **Origin header check:** Next.js compares `Origin` header to `Host` header — aborts if they don't match

**No additional CSRF library is needed** for this app. The built-in protection is sufficient for an admin-only, single-origin application.

**If deployed behind a reverse proxy:** Add to `next.config.ts`:
```typescript
experimental: {
  serverActions: {
    allowedOrigins: ['chemosystem.vercel.app'],
  },
},
```

### Pattern 5: `server-only` Guard for DAL

**Source:** [Next.js Data Security Guide](https://nextjs.org/docs/app/guides/data-security) — HIGH confidence

`src/lib/dal.ts` currently has NO `import 'server-only'` guard. Any developer could accidentally import it in a client component and expose session verification logic to the browser bundle.

```typescript
// Add at the top of src/lib/dal.ts

```

### Pattern 6: `NEXT_SERVER_ACTIONS_ENCRYPTION_KEY` Environment Variable

**Source:** [Next.js Data Security Guide — Closures section](https://nextjs.org/docs/app/guides/data-security) — HIGH confidence

Server Actions that capture variables in closures encrypt those variables. Without a fixed key, each deployment gets a different key, causing "Server Action expired" errors after re-deployment.

```bash

# Add to .env.local and Vercel env vars:
NEXT_SERVER_ACTIONS_ENCRYPTION_KEY="<base64-32-bytes>"
```

## Supabase RLS Audit

### Current State Assessment

**Source:** Direct codebase audit of `00002_rls_policies.sql` and `00012_access_control.sql`

All tables have RLS enabled. Current policies are "permissive authenticated" — any authenticated user can read/write.

**For the current context (Sharon only):** This is acceptable. RLS is acting as a safety net (database-level) while the primary access control is at the Server Action level via `verifySession()`.

**One genuine risk:** `user_permissions` INSERT/UPDATE policies use `WITH CHECK (true)` — meaning any authenticated user can grant themselves admin-level permissions by directly calling the DB (bypassing Server Actions). This is the most important RLS hardening target.

**Recommended additions (migration 00013):**

```sql
-- Tighten user_permissions writes to admin-only
-- Drop permissive policies
DROP POLICY IF EXISTS "user_permissions_insert" ON user_permissions;
DROP POLICY IF EXISTS "user_permissions_update" ON user_permissions;
DROP POLICY IF EXISTS "user_permissions_delete" ON user_permissions;

-- Replace with admin-only write policies
CREATE POLICY "user_permissions_insert_admin" ON user_permissions
  FOR INSERT TO authenticated
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM users
      WHERE auth_user_id = auth.uid()
        AND is_admin = true
        AND deleted_at IS NULL
    )
  );

CREATE POLICY "user_permissions_update_admin" ON user_permissions
  FOR UPDATE TO authenticated
  USING (true)
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM users
      WHERE auth_user_id = auth.uid()
        AND is_admin = true
        AND deleted_at IS NULL
    )
  );

CREATE POLICY "user_permissions_delete_admin" ON user_permissions
  FOR DELETE TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM users
      WHERE auth_user_id = auth.uid()
        AND is_admin = true
        AND deleted_at IS NULL
    )
  );
```

## Common Pitfalls

### Pitfall 1: Edge Runtime Rate Limiting (Doesn't Work)
**What goes wrong:** Developer puts rate limiting Map in `proxy.ts` (middleware).
**Why it happens:** `proxy.ts` runs on Edge Runtime — no persistent memory. The Map is empty on every request.
**How to avoid:** Rate limiting MUST be in the login Server Action (Node.js runtime), not in `proxy.ts`.
**Warning signs:** Rate limiter never blocks anything no matter how many attempts.

### Pitfall 2: CSP Blocks Supabase Storage URLs
**What goes wrong:** CSP `connect-src 'self'` blocks Supabase API calls. Photos from Supabase Storage fail to load.
**Why it happens:** Supabase URL is a different origin from the app.
**How to avoid:** Add `NEXT_PUBLIC_SUPABASE_URL` to both `connect-src` and `img-src` in CSP.
**Warning signs:** Browser console shows CSP violation for `*.supabase.co`.

### Pitfall 3: HSTS on Development
**What goes wrong:** HSTS header causes browser to refuse HTTP connections in development.
**Why it happens:** `max-age=63072000` tells browser to always use HTTPS.
**How to avoid:** Apply HSTS header only in production — check `NODE_ENV`.
**Warning signs:** Localhost stops working after HSTS header is served.

### Pitfall 4: CSP Breaks Tailwind v4 / shadcn/ui
**What goes wrong:** `style-src 'self'` without `'unsafe-inline'` breaks Tailwind v4 and shadcn/ui which inject inline styles.
**Why it happens:** Tailwind v4 uses a different CSS injection strategy than v3.
**How to avoid:** Include `'unsafe-inline'` in `style-src` for Tailwind v4. Nonce-based approach is an option but disruptive.
**Warning signs:** All component styles disappear after CSP is added.

### Pitfall 5: X-Frame-Options Deprecated in Favor of CSP
**What goes wrong:** Only setting `X-Frame-Options: DENY` but not `frame-ancestors 'none'` in CSP.
**Why it happens:** X-Frame-Options is the older mechanism; CSP `frame-ancestors` supersedes it in modern browsers.
**How to avoid:** Set BOTH `X-Frame-Options: DENY` AND `frame-ancestors 'none'` in CSP for full coverage.
**Warning signs:** Modern browsers still allow framing if only X-Frame-Options is set (Chrome ignores it when CSP frame-ancestors is present).

### Pitfall 6: `audit_log` INSERT Policy Too Permissive
**What goes wrong:** `audit_log_insert` policy uses `WITH CHECK (true)` — any authenticated user can write arbitrary audit records.
**Why it happens:** Phase 1 simplicity — not a priority when only one user exists.
**How to avoid:** For now acceptable. When ChemoSys is built with multiple users, tighten to `WITH CHECK (auth.uid() = user_id::uuid)` equivalent.

### In-Memory Rate Limiter for Login Server Action

```typescript
// Source: Verified pattern — in Node.js runtime (not Edge), module-level Map persists
// Caveat: resets on cold start; not shared across multiple serverless instances
// Acceptable for: single admin user, single Vercel instance deployment

type RateLimitEntry = { count: number; resetAt: number }
const loginAttempts = new Map<string, RateLimitEntry>()
const RATE_LIMIT_MAX = 5
const RATE_LIMIT_WINDOW_MS = 60_000 // 1 minute

function getClientIp(headersList: Headers): string {
  // Vercel sets x-forwarded-for; direct connection uses x-real-ip
  return (
    headersList.get('x-forwarded-for')?.split(',')[0]?.trim() ??
    headersList.get('x-real-ip') ??
    'unknown'
  )
}

function checkLoginRateLimit(ip: string): { allowed: boolean; remainingMs?: number } {
  const now = Date.now()
  const entry = loginAttempts.get(ip)

  if (!entry || now > entry.resetAt) {
    loginAttempts.set(ip, { count: 1, resetAt: now + RATE_LIMIT_WINDOW_MS })
    return { allowed: true }
  }

  if (entry.count >= RATE_LIMIT_MAX) {
    return { allowed: false, remainingMs: entry.resetAt - now }
  }

  entry.count++
  return { allowed: true }
}

// Usage in login() action:
import { headers } from 'next/headers'

export async function login(prevState, formData) {
  const headersList = await headers()
  const ip = getClientIp(headersList)

  const { allowed } = checkLoginRateLimit(ip)
  if (!allowed) {
    return { error: 'יותר מדי ניסיונות התחברות. נסה שוב בעוד דקה.' }
  }
  // ... existing login logic
}
```
