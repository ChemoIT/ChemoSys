# Phase 03.1 Plan 03: Security Hardening — RLS Admin-Only Write Policies Summary

**RLS policies on user_permissions tightened from permissive (any authenticated) to admin-only via is_admin=true EXISTS check, closing the Broken Access Control gap where non-admin users could self-grant permissions by direct DB access.**

- Dropped 4 permissive policies (user_permissions_insert, user_permissions_update, user_permissions_delete from 00002, plus user_permissions_update from 00012)

## Decisions Made

- **Policy naming:** New policies use `_admin` suffix (e.g., `user_permissions_insert_admin`) to avoid naming conflicts in case `DROP POLICY IF EXISTS` fails silently. This means pg_policies will show a clean set of 4 distinctly-named policies.
- **SELECT left permissive:** The existing `user_permissions_select` from 00002 was not modified. Reads are harmless — the real protection layer is `get_user_permissions()` which is SECURITY DEFINER. Keeping SELECT permissive avoids breaking any direct table queries that may be used in admin debug scenarios.
- **UPDATE USING (true):** Consistent with Pitfall 9 pattern from 00002 — UPDATE policy uses `USING (true)` so rows can always be matched for update; the `WITH CHECK` clause is where the admin restriction lives.
- **Bootstrap admin unaffected:** Sharon's admin operations via `createAdminClient()` (service role key) bypass RLS entirely, so these new restrictions do not break any admin-shell functionality.
